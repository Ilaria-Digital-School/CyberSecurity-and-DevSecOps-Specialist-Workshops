# Capturing a TCP handshake with Wireshark

**Theoretical Recap** The TCP three-way handshake is the mechanism by which two hosts establish a reliable connection before any data is exchanged. It consists of three segments: SYN (client initiates, proposes an Initial Sequence Number), SYN-ACK (server acknowledges and proposes its own ISN), and ACK (client acknowledges the server's ISN). This exchange happens at Layer 4 of the OSI model (Transport), but Wireshark lets you observe it all the way down to the Ethernet frame at Layer 2. A key mental model: think of the handshake as a mutual introduction where both sides agree on 'conversation IDs' (sequence numbers) before exchanging a single byte of payload. Common mistakes beginners make include confusing the SYN-ACK with a data packet, missing the handshake because capture was started too late, or failing to apply the right display filter and getting overwhelmed by noise. A quick example: when your browser connects to a web server on port 443, Wireshark will show a frame flagged [SYN] from your IP to the server, immediately followed by [SYN, ACK] from the server back to you, then your [ACK] closing the ritual. From that point on, TLS negotiation begins on top.
