# Workshop: installing and verifying the lab virtual machine

**Theoretical Recap** A virtual machine (VM) is a software emulation of a physical computer. It runs an operating system and applications in an isolated environment, completely separated from your host system. This isolation is fundamental in cybersecurity and DevSecOps: it allows you to perform penetration testing, experiment with vulnerable applications, and manipulate potentially dangerous tools without risking your main system or violating any legal boundaries. The key mental model to internalize is this: your host machine is your trusted workspace; your VM is your controlled sandbox. Everything risky, experimental, or potentially destructive stays inside the VM. A snapshot is your safety net — a frozen state of the VM you can restore instantly if something breaks. Hypervisors (VirtualBox, VMware, Hyper-V) are the software layer that manages this virtualization. Common mistakes at this stage include: skipping snapshot creation before starting sensitive work, allocating too little RAM or disk space to the VM (causing slow or failed installations), ignoring network adapter configuration (which determines whether your VM can reach the internet, communicate with other VMs, or remain fully isolated), and neglecting to verify that virtualization is enabled in the BIOS/UEFI. A practical mini example: a learner installs Kali Linux as a VM, allocates only 1 GB of RAM — the system becomes unusable. The fix: reallocate at least 2 GB (4 GB recommended) before first boot.
