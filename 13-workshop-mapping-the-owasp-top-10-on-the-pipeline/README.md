# Workshop: mapping the OWASP Top 10 on the pipeline

**Theoretical Recap** The OWASP Top 10 is a regularly updated reference document listing the ten most critical web application security risks. It is not a checklist to mechanically tick off, but a risk map that helps teams prioritize where to focus their security efforts. Each entry represents a category of vulnerability (e.g., A01 Broken Access Control, A02 Cryptographic Failures, A03 Injection, A05 Security Misconfiguration, A06 Vulnerable and Outdated Components, etc.) with associated attack scenarios and remediation guidance. A CI/CD pipeline is a sequence of automated stages: source code management (Git), static analysis (SAST), dependency scanning (SCA), build, dynamic testing (DAST), containerization, and deployment. Each stage is an opportunity — or a failure point — relative to one or more OWASP categories. For example, A06 (Vulnerable and Outdated Components) maps directly to SCA tooling in the pipeline, while A03 (Injection) can be caught early by SAST rules during the commit or build phase. A common mistake is treating the OWASP Top 10 as a pure developer concern, ignoring the pipeline as a detection and enforcement layer. Another frequent error is mapping each risk to only one stage, when in reality most risks should be addressed at multiple points (defense in depth). Keep this in mind: the pipeline is your automated security net, and the OWASP Top 10 tells you what that net must catch. Mini example: A developer commits code containing a raw SQL query. A SAST tool configured at the pre-commit or CI stage flags it as a potential injection...
