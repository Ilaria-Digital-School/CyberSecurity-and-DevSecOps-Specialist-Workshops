# Workshop: simulating a committed secret and correcting it

**Theoretical Recap** In the world of DevSecOps, one of the most common and dangerous mistakes a developer can make is accidentally committing a secret — a password, an API key, a database connection string, a private token — directly into a Git repository. Once a secret is committed, it becomes part of the repository's history, even if you delete it in a subsequent commit. This is a critical point: deleting a file or a line in Git does not erase the past. Anyone with access to the repository (and sometimes even without, if the repo is public) can run `git log`, `git show`, or `git diff` to retrieve the exposed data. A common mental model: think of Git history as a ledger. Every entry is permanent unless you explicitly rewrite history using commands like `git rebase -i`, `git filter-branch`, or the modern `git filter-repo` tool. Simply adding a new commit to 'fix' the secret leaves the original commit fully intact and readable. Common mistakes include: believing that a `.gitignore` added after the fact will retroactively protect already-tracked files; thinking that making a repository private after a leak is sufficient (the secret may already have been scraped by bots); and forgetting that CI/CD pipelines, forks, or clones may already hold copies of the exposed history. Mini example: a developer commits a file `config.py` containing `API_KEY = 'sk-prod-abc123xyz'`. Even after removing the key and committing again, running `git log -p` will reveal the original value in the diff of the first commit....
