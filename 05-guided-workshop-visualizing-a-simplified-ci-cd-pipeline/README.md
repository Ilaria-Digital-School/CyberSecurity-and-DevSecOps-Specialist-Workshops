# Guided workshop: visualizing a simplified CI/CD pipeline

**Theoretical Recap** CI/CD stands for Continuous Integration and Continuous Delivery (or Deployment). It is the backbone of modern DevOps — and by extension, DevSecOps — workflows. Continuous Integration means that every time a developer pushes code to a shared repository, an automated process kicks in: the code is built, tested, and validated without human intervention. Continuous Delivery ensures that this validated code is always in a deployable state, ready to be shipped to a staging or production environment at any moment. Continuous Deployment goes one step further: every validated change is automatically deployed to production. A simplified CI/CD pipeline typically flows through these stages: Code → Build → Test → Security Scan → Deploy → Monitor. Each stage acts as a gate: if a stage fails, the pipeline stops and the team is alerted. This is exactly where DevSecOps enters the picture — security checks (SAST, DAST, SCA) are embedded inside the pipeline rather than bolted on at the end. A common beginner mistake is to confuse Continuous Delivery with Continuous Deployment. Remember: Delivery means the code is ready and could be deployed (human approval may still be required), while Deployment means it is automatically pushed live with no manual step. Another frequent confusion is treating the pipeline as a linear script when in reality it is a series of automated jobs that can run in parallel or conditionally. Mini example: a developer commits a login feature → the pipeline builds the application → unit tests run → a SAST...
