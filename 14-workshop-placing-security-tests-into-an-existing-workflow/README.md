# Workshop: placing security tests into an existing workflow

**Theoretical Recap** Integrating security tests into an existing CI/CD workflow is the operational heart of DevSecOps. The core principle is simple: security checks must not be an afterthought bolted onto the end of a pipeline — they must be embedded at every relevant stage. Three families of tests matter here. SAST (Static Application Security Testing) analyzes source code without executing it, catching issues like hardcoded secrets, insecure patterns, or dangerous imports early in the pipeline, ideally at the commit or pull-request stage. DAST (Dynamic Application Security Testing) interacts with a running application, simulating an attacker's perspective to find runtime vulnerabilities such as injection flaws or broken authentication — it belongs in staging environments. SCA (Software Composition Analysis) scans third-party dependencies for known CVEs and license issues, and should run whenever dependencies change. A common mistake is placing all security gates at the very end of the pipeline, turning them into release blockers that slow delivery without providing early feedback. Another mistake is treating every finding as a critical blocker: triage and severity thresholds are essential so that low-severity findings generate warnings without stopping the build. The mental model to retain: map each test type to the pipeline stage where it delivers the best signal-to-noise ratio and the lowest cost to fix.
