# Filter UDP DNS traffic

**Theoretical Recap** DNS (Domain Name System) is the protocol responsible for translating human-readable domain names (e.g., google.com) into IP addresses that machines can route. By default, DNS operates over UDP on port 53 — a deliberate design choice prioritizing speed over reliability, since name resolution queries are small and latency-sensitive. Unlike TCP, UDP is connectionless: there is no handshake, no acknowledgment, and no guaranteed delivery. This makes DNS traffic lightweight but also a prime vector for attacks such as DNS spoofing, amplification DDoS, and data exfiltration via DNS tunneling. In Wireshark, DNS traffic is easy to identify because it always targets or originates from port 53, uses the UDP transport layer, and carries a structured payload that Wireshark decodes automatically with its built-in DNS dissector. A common beginner mistake is confusing DNS over TCP (used for large responses or zone transfers) with the standard UDP variant — always check the transport layer before drawing conclusions. Another frequent error is writing overly broad filters that capture all UDP traffic rather than isolating DNS specifically. Mental model: think of a DNS query as a sticky note you send to a directory service asking "what's the address for X?" — UDP delivers the note fast without waiting for a receipt. Wireshark lets you intercept and read every such note passing through your network interface. Mini example: the BPF filter `udp port 53` captures only UDP packets on port 53, which in practice means DNS queries and responses — nothing more, nothing less. ---...
