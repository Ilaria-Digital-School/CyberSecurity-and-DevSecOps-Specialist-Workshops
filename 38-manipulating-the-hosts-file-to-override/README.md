# Manipulating the hosts file to override

**Theoretical Recap** The hosts file is one of the oldest name resolution mechanisms in computing, predating DNS entirely. Located at `/etc/hosts` on Linux/macOS and `C:\Windows\System32\drivers\etc\hosts` on Windows, it acts as a local, static lookup table that maps hostnames to IP addresses. When your system resolves a domain name, the OS typically checks the hosts file *before* querying any DNS server — this priority order is defined in `/etc/nsswitch.conf` on Linux (look for the `hosts:` line, usually reading `files dns`). This means that any entry in your hosts file will override whatever DNS would have returned, silently and completely. A common mental model: think of the hosts file as a local DNS shortcut that always wins. If you add `127.0.0.1 google.com` to your hosts file, your browser will connect to your own machine when you visit google.com, regardless of what the real DNS says. Common mistakes include forgetting to flush the DNS cache after editing the file (on Linux: `sudo systemd-resolve --flush-caches`), editing the file without root privileges (the changes simply won't save), or expecting the override to affect other machines on the network (it is strictly local).
