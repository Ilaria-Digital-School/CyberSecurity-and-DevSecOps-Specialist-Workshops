# Workshop: extracting key information from the report

**Theoretical Recap** A vulnerability report is a structured document that communicates security findings to both technical and non-technical stakeholders. Understanding how to read and extract key information from such a report is a foundational skill for any DevSecOps specialist. A well-formed report typically contains: an executive summary (high-level risk overview), a findings section (individual vulnerabilities with details), a severity rating (using scales like CVSS or qualitative labels: Critical, High, Medium, Low, Informational), a description of the attack vector and potential impact, and remediation recommendations. The most common mistake beginners make is focusing exclusively on the severity score without reading the context — a Medium vulnerability in a public-facing authentication endpoint can be far more dangerous than a High vulnerability in an internal tool with no network exposure. Another frequent error is ignoring the 'Affected Component' field, which maps directly to what needs to be patched in your pipeline or codebase. A useful mental model: treat each finding as a mini-story with four questions — What is broken? Where is it? What can an attacker do with it? How do we fix it? If you can answer all four from the report entry, you have extracted the essential information.
