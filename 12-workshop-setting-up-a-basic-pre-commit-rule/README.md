# Workshop: setting up a basic pre-commit rule

**Theoretical Recap** A pre-commit hook is a script that Git executes automatically before finalizing a commit. It acts as a last-line-of-defence gate: if the script exits with a non-zero code, the commit is rejected and the developer is forced to fix the problem before the code ever reaches the repository. This mechanism is part of the broader family of Git hooks stored in the hidden `.git/hooks/` directory of every repository. The mental model is simple: think of a pre-commit hook as a security guard standing at the entrance of your codebase. Every time someone tries to push new content in, the guard runs a checklist. If anything on the checklist fails — a hardcoded API key, a password in plain text, a forbidden file extension — the guard blocks the entry and explains why. A common mistake beginners make is confusing pre-commit hooks with CI/CD pipeline checks. Hooks are local and client-side by default; they can be bypassed with `git commit --no-verify`. This is why hooks are a first barrier, not the only barrier. Another frequent error is writing a hook script without making it executable (`chmod +x`), which silently renders it inactive.
