# Guided workshop: placing security checks in an imaginary pipeline

THEORETICAL RECAP A CI/CD pipeline (Continuous Integration / Continuous Delivery) is the automated chain that takes source code from a developer's commit all the way to a running application in production. In a DevSecOps approach, security is not a final gate — it is embedded at every stage of this chain, as early as possible. This is the 'shift left' principle: catch vulnerabilities when they are cheapest to fix, i.e., before the code is even deployed. Three families of automated security checks are fundamental here. SAST (Static Application Security Testing) analyzes source code or binaries without executing them, looking for insecure coding patterns, hardcoded secrets, or dangerous function calls. DAST (Dynamic Application Security Testing) attacks a running version of the application from the outside, simulating what a real attacker would do, uncovering runtime vulnerabilities like injection flaws or broken authentication. SCA (Software Composition Analysis) inventories the open-source libraries and dependencies your project uses and flags known CVEs (Common Vulnerabilities and Exposures) against them. A common mistake beginners make is placing all security checks only at the end of the pipeline, treating them like a checkbox before release. This defeats the purpose: a SAST scan run only before deployment means developers have already built layers of code on top of a flaw. Another frequent error is confusing SAST and DAST — remember: SAST needs code, DAST needs a live application.
