# Read a captured Ethernet frame

THEORETICAL RECAP An Ethernet frame is the fundamental unit of data at Layer 2 (Data Link) of the OSI model. When two devices communicate on a local network, data is broken down into frames before being transmitted over the physical medium. Each Ethernet frame follows a strict structure: a Preamble (7 bytes) used for synchronization, a Start Frame Delimiter or SFD (1 byte), a Destination MAC Address (6 bytes), a Source MAC Address (6 bytes), an EtherType/Length field (2 bytes) indicating which Layer 3 protocol is encapsulated (e.g., 0x0800 for IPv4, 0x0806 for ARP, 0x86DD for IPv6), a Payload (46 to 1500 bytes) containing the actual data passed down from upper layers, and a Frame Check Sequence or FCS (4 bytes) used for error detection via CRC. A common mistake beginners make is confusing MAC addresses with IP addresses. MAC addresses are hardware identifiers burned into network interface cards and operate at Layer 2 — they are used for local delivery only. IP addresses operate at Layer 3 and handle routing between networks. Another frequent error is overlooking the EtherType field, which is the key to understanding what protocol lives inside the payload. Mini example: if you capture a frame with EtherType 0x0800, you know the payload contains an IPv4 packet. If you then look inside that packet, you might find a TCP segment, and inside that segment, an HTTP request. This is encapsulation in action — each layer wraps the one above it.
