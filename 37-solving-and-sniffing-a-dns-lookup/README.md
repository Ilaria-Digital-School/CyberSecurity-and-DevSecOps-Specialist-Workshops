# Solving and sniffing a DNS lookup

**Theoretical Recap** DNS (Domain Name System) is the internet's phone book: it translates human-readable hostnames (e.g., example.com) into IP addresses that routers and hosts can use. A DNS lookup follows a strict hierarchical resolution chain: your machine first checks its local cache, then queries a recursive resolver (usually provided by your ISP or configured manually, e.g., 8.8.8.8), which in turn contacts root servers, TLD servers, and finally the authoritative nameserver for the domain. The result is cached according to the TTL (Time To Live) value attached to each DNS record. Key record types to know: A (IPv4 address), AAAA (IPv6 address), CNAME (alias), MX (mail exchange), NS (nameserver), PTR (reverse lookup), TXT (arbitrary text, used for SPF/DKIM). DNS operates over UDP port 53 by default (TCP for large responses or zone transfers). Common mistakes: confusing the recursive resolver with the authoritative nameserver; forgetting that cached results may be stale; ignoring that DNS is unencrypted by default (hence interceptable). From a security standpoint, DNS is a prime target for reconnaissance, cache poisoning, and data exfiltration via DNS tunneling. Mini example: running `dig google.com` returns the A record with its TTL. Running the same command twice in quick succession shows the TTL decreasing, confirming local caching behavior. --- **Workshop Brief** This workshop puts into practice the DNS concepts introduced in the previous chapter.
