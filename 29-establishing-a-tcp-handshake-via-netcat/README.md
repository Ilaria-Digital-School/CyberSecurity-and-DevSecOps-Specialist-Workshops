# Establishing a TCP handshake via netcat

**Theoretical Recap** The TCP (Transmission Control Protocol) three-way handshake is the foundational mechanism by which two hosts establish a reliable connection before any data exchange occurs. It operates at Layer 4 (Transport) of the OSI model and consists of three steps: SYN, SYN-ACK, and ACK. The client initiates by sending a SYN packet, the server responds with SYN-ACK to acknowledge and propose its own sequence number, and finally the client confirms with an ACK. Only after this exchange is the connection considered established. A common misconception is confusing connection establishment with data transfer — the handshake itself carries no application payload. Another frequent mistake is thinking UDP behaves the same way: UDP is connectionless and performs no handshake at all, which is why it is faster but unreliable. Understanding this distinction is critical in security contexts: many attacks (SYN flood, for example) specifically exploit the handshake mechanism. Netcat (nc) is often called the 'Swiss Army knife' of networking. It can open raw TCP connections, act as a server or client, and let you observe exactly what happens at the transport layer. A simple mental model: netcat on one terminal acts as the server listening on a port; netcat on another terminal acts as the client connecting to it — the OS TCP stack performs the full handshake transparently between them.
