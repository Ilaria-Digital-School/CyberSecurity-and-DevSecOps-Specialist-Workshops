# Workshop: Writing a micro-report on an imaginary bug

THEORETICAL RECAP A vulnerability report is the primary deliverable of any security assessment. It transforms technical findings into actionable intelligence for developers, managers, and security teams. A well-structured report answers four essential questions: What is the vulnerability? Where exactly does it exist? What is the realistic impact? How should it be fixed? The core components of any vulnerability report entry are: (1) a clear, descriptive title; (2) a severity rating using a recognized scale such as Critical / High / Medium / Low / Informational; (3) a concise description of the flaw; (4) the affected component or endpoint; (5) the steps to reproduce the issue; (6) the proof of concept (evidence); (7) the business impact, framed through the CIA triad (Confidentiality, Integrity, Availability); and (8) a concrete remediation recommendation. Common mistakes beginners make: writing vague titles like 'SQL injection found' instead of 'Unauthenticated SQL Injection on /api/login allows full database dump'; conflating description with reproduction steps; ignoring the business impact entirely; and recommending remediation without specificity (e.g., 'fix the code' is not a recommendation). Mini example: Title — 'Reflected XSS on /search endpoint exposes session tokens'. Severity — High. Description — The search parameter is rendered unsanitized in the HTML response, allowing script injection. Steps to reproduce — Submit ?q=alert(document.cookie) in the search bar. CIA Impact — Confidentiality: session tokens can be stolen. Integrity: DOM can be manipulated. Remediation — Encode all user-supplied output using context-aware escaping before rendering.
