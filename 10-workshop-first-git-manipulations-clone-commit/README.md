# Workshop: first Git manipulations (clone, commit)

THEORETICAL RECAP Git is a distributed version control system that tracks changes in files over time. Every project managed with Git lives inside a repository (repo), which stores the full history of modifications. Understanding a few core concepts is essential before touching the command line. A clone is a full local copy of a remote repository. When you clone a repo, you download its entire history, branches, and files onto your machine. From that moment, your local copy is independent but can synchronize with the remote (called 'origin') at any time. A commit is a snapshot of your working directory at a specific point in time. Think of it as a labelled save point. Each commit carries a unique hash (SHA-1), a timestamp, the author's identity, and a message explaining what changed. A good commit message is short, imperative, and meaningful — 'Add authentication module' is good; 'stuff' is not. The mental model: remote repo → clone → local repo → edit files → stage changes (git add) → commit (git commit) → push back to remote (git push). Common mistakes beginners make: committing without a meaningful message, forgetting to stage files before committing (nothing gets committed without git add), and confusing the working directory with the staging area (index). Another frequent error is cloning with HTTPS when SSH keys are already configured, causing authentication loops.
