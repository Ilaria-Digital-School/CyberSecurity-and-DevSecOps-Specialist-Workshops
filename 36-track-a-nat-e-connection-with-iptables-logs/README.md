# Track a NATée connection with iptables logs

**Theoretical Recap** Network Address Translation (NAT) is a mechanism that rewrites IP headers at the boundary between a private network and the public internet. When a packet leaves your LAN, the router replaces the source private IP (e.g., 192.168.1.10) with its own public IP and records the mapping in a state table so return traffic can be forwarded back to the right host. This is called SNAT (Source NAT) or masquerading. DNAT (Destination NAT) works in reverse: inbound packets destined for a public IP are redirected to a private host. iptables is the classic Linux firewall and packet-filtering tool that operates at the netfilter level inside the kernel. It organises rules into tables (filter, nat, mangle) and chains (PREROUTING, POSTROUTING, INPUT, OUTPUT, FORWARD). The LOG target in iptables does not drop packets; it simply writes a kernel log entry (visible via dmesg or /var/log/syslog / /var/log/kern.log) containing metadata: source IP, destination IP, port, interface, TTL, and more. A common mistake beginners make is inserting a LOG rule after a DROP rule — the packet is already discarded, so the log entry is never written. Always place LOG rules before terminating rules. Another pitfall: forgetting that NAT rewrites happen at different chain stages, so the IP you see in FORWARD may already be translated compared to what you see in PREROUTING. Mini example: a packet from 192.168.1.5:4321 reaches the router, PREROUTING logs show the original source; after MASQUERADE in POSTROUTING the source becomes 203.0.113.1:some_ephemeral_port.
