# Workshop: Adapting a secure pipeline to an e-commerce project

THEORETICAL RECAP A CI/CD pipeline (Continuous Integration / Continuous Delivery) is an automated sequence of steps that takes source code from a developer's commit all the way to a production deployment. In a standard DevOps pipeline, the typical stages are: code commit → build → automated tests → artifact storage → deployment to staging → deployment to production. DevSecOps extends this by embedding security checks at every stage rather than treating security as a final gate. The three pillars you must keep in mind are: SAST (Static Application Security Testing), which analyzes source code before it runs to catch vulnerabilities early; DAST (Dynamic Application Security Testing), which attacks a running application to find runtime vulnerabilities; and SCA (Software Composition Analysis), which scans third-party libraries and dependencies for known CVEs. A common beginner mistake is placing all security checks at the end of the pipeline — this is the 'security as a bolt-on' anti-pattern. Another frequent error is forgetting to protect secrets (API keys, database credentials) inside the pipeline itself using a secrets manager or environment variables. Mini example: imagine a basic pipeline for a blog app. You add a SAST scan (e.g., Semgrep) right after the build step, an SCA scan (e.g., OWASP Dependency-Check) before artifact packaging, and a DAST scan (e.g., OWASP ZAP) against the staging environment before production deployment. Each security gate can block the pipeline if a critical finding is detected. This is the mental model you will now apply to a more complex, realistic context. ---...
